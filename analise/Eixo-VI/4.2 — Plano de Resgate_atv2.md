# 4.2 — Plano de Resgate Sugerido

## 4.2.1 Refatoração Conceitual – Chain of Responsibility na Execução de Ferramentas

A auditoria identificou que o agente do Vanna 2.0 executa chamadas de ferramentas de forma **cega**, sem nenhuma camada de validação entre a decisão do LLM e a execução real no banco de dados.

**Código violado:**  
`src/vanna/core/agent/agent.py` — método `_send_message()`  
Linha crítica: `result = await self.tool_registry.execute(tool_call, context)`

**Padrão aplicado:** Chain of Responsibility (implementado como Middlewares/Interceptors de Execução)

**Justificativa arquitetural:**  
O repositório já utiliza Chain of Responsibility/Middleware na camada de comunicação com LLMs (`src/vanna/core/middleware` → classe `LlmMiddleware`). No entanto, houve uma falha de design ao **não estender esse padrão para a camada de execução de ferramentas** (`ToolRegistry` e `Agent`). Nossa proposta corrige essa dívida técnica criando um `ExecutionMiddleware`.

**Fluxo proposto:**

| Etapa | Componente | Ação |
|-------|-----------|------|
| 1 | `Agent._send_message()` | Recebe `tool_call` do LLM |
| 2 | `ToolExecutionChain` | Inicia a corrente de middlewares |
| 3 | `SQLSanitizationMiddleware` | Bloqueia DROP/DELETE não autorizados |
| 4 | `RateLimitMiddleware` | Controla frequência de chamadas a APIs pagas |
| 5 | `LoggingMiddleware` | Registra auditoria de cada execução |
| 6 | `tool_registry.execute()` | Executa **apenas se toda a corrente aprovar** |

**Resultado:** O método `await self.sql_runner.run_sql()` nunca é chamado se a corrente identificar um comando perigoso — a IA não consegue causar dano mesmo em caso de alucinação.

## 4.2.2 Roadmap de Maturidade – Alinhamento ao Nível G do MPS.BR

O MPS.BR define sete níveis de maturidade (G → A). O **Nível G**, ponto de entrada, exige a institucionalização de **GPR** (Gerência de Projetos) e **GRE** (Gerência de Requisitos). A análise do repositório evidenciou que o projeto opera de forma informal, sem processos definidos — equivalente ao **pré-Nível G**.

As três ações abaixo formam o roadmap para alcançar o Nível G em até **90 dias**.

---

### AÇÃO 1 — Institucionalizar a Gerência de Projetos (GPR) via GitHub Projects
**MPS.BR:** GPR | **Prazo:** Semanas 1–3 | **Prioridade:** CRÍTICA

A auditoria (Eixo A.3) detectou ritmo irregular com picos de *crunch* e concentração excessiva em um único mantenedor (~6× mais commits que o 2º colaborador) — risco classificado como **ALTO**.

**Ações concretas:**
- Criar **GitHub Project Board** (Backlog → In Progress → Review → Done) com milestones mensais
- Adotar **Conventional Commits** (feat, fix, docs, chore) para CHANGELOG automatizado
- Exigir **aprovação de 2 revisores** em PRs que afetem módulos críticos (`core/agent`, `core/recovery`, `legacy/base`)
- **Rotação de mantenedores**: designar 2 co-maintainers com acesso de merge
- Criar **Registro de Riscos** no wiki com: impacto, probabilidade e estratégia de mitigação

**Resultado esperado (GPR):** Planejamento documentado, estimativas registradas e monitoramento explícito de desvios.

---

### AÇÃO 2 — Implementar Gerência de Requisitos (GRE) com Templates Estruturados
**MPS.BR:** GRE | **Prazo:** Semanas 2–5 | **Prioridade:** ALTA

A auditoria (Eixo A.1) mostrou que decisões técnicas são tomadas informalmente nos comentários de issues, sem RFCs. A issue #659 evidenciou mudança de escopo não documentada — violação direta dos critérios da GRE.

**Ações concretas:**
- Criar **Issue Templates** (`.github/ISSUE_TEMPLATE/`) com campos obrigatórios: Descrição, Critério de Aceitação, Impacto Arquitetural, Módulos Afetados
- Adotar formato **User Story**: *"Como [papel], quero [funcionalidade] para [benefício]"*
- Implementar **labels de rastreabilidade**: `req:functional`, `req:non-functional`, `arch:breaking`, `risk:high/medium/low`
- Criar **ARCHITECTURE.md** com ADRs (Architecture Decision Records)
- **Branch protection**: todo PR deve referenciar ao menos uma issue rastreável antes do merge

**Resultado esperado (GRE):** Requisitos documentados com bidirecionalidade entre requisitos e implementações.

---

### AÇÃO 3 — Elevar a Qualidade via Pipeline CI/CD e Testes Automatizados
**MPS.BR:** Qualidade do Produto e Processo | **Prazo:** Semanas 3–8 | **Prioridade:** ALTA

A auditoria (Eixos A.1, B.2, C.3) evidenciou: funcionalidades integradas sem testes (caso PGVector), `VannaBase` com 2.125 linhas sem testes unitários isolados e agente executando `tool_calls` sem validação — vulnerabilidade de SQL injection via alucinação do LLM.

**Ações concretas:**
- **GitHub Actions** obrigatório em todo PR: lint (flake8/ruff) → testes (pytest) → cobertura mínima 70% (coverage.py)
- **Testes de contrato** para cada integração LLM: garantir que `OpenAILlmService`, `AnthropicLlmService` e `OllamaLlmService` retornam o mesmo schema `LlmResponse`
- **Testes de segurança** para o `ExecutionMiddleware`: payloads com `DROP TABLE`, `DELETE FROM` verificando bloqueio antes do banco
- **SAST com Bandit** integrado ao CI, detectando padrões de SQL injection automaticamente
- Estabelecer **Definition of Done (DoD)**: toda feature só é pronta com testes, documentação e aprovação de revisor

**Resultado esperado:** Pipeline automatizado que previne regressões, reduz dívida técnica e aumenta confiança na evolução do sistema.

---

| Semana | Ação | Processo MPS.BR |
|--------|------|----------------|
| 1–3 | GitHub Projects + Conventional Commits + Rotação de mantenedores | GPR |
| 2–5 | Issue Templates + User Stories + ARCHITECTURE.md | GRE |
| 3–8 | CI/CD + Testes de contrato + SAST + Definition of Done | Qualidade |
